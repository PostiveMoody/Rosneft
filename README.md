# Техническое задание
Перед Вами стоит задача разработать небольшое SPA приложение, которое автоматизирует процесс управления жизненным циклом обращений (заявки на выполнение каких-либо работ). Обращения используются в приложении как способ решить какую-либо проблему пользователя, возникшую в ходе работы в компании или отдельного подразделения. При разработке необходимо использовать следующие технологии: 

  •	Бэкэнд - Asp.Net Core, Entity Framework Core, MS SQL

# User Story - 1: Пользователь может создать новое обращение в системе
Задачи: 
Разработать детальное представление (карточку) обращения. 
Требования: 
1.	Карточка обращения должна содержать следующие поля:

    a.	Id — уникальный идентификатор обращения. Поле должно заполняться автоматически после сохранения нового обращения.

    b.	Инициатор — физическое лицо, зарегистрировавшее обращение. Текстовое поле. Ограничение поля 150 буквенных символов. Допустимые символы для ввода: латиница, кириллица. Недопустимые символы для ввода: арабские цифры. Обязательное поле.
  
    c.	Название — тема обращения, кратко описывающая возникшую проблему. Текстовое поле. Недопустимые символы для ввода: арабские цифры. Обязательное поле.
  
    d.	Описание — развернутое описание проблемы. Текстовое поле. Допустимые символы для ввода: буквенные символы на латинице и кириллице, арабские цифры, спецсимволы. Ограничение: до 1000 символов. Обязательное поле.
  
    e.	Крайний срок взятия в работу — формат ДД.ММ.ГГГГ ЧЧ:ММ:СС.
  
    f.	Статус — поле может принимать фиксированные значения: «Новое», «В работе», «Завершено», «Просрочено». При создании обращения поле должно по умолчанию автоматически заполняться значением «Новое». 
  
    g.	Категория — поле может принимать фиксированные значения из: «Консультация», «Жалоба», «Инцидент», «Другое». Заполняется при помощи выбора из выпадающего списка. Необязательное поле (не должно быть помечено никакими спецсимволами). 
  
    h.	Дата создания - формат ДД.ММ.ГГГГ ЧЧ:ММ:СС. Заполняется автоматически при создании обращения.
  
  2.	Карточка обращения должна содержать следующие кнопки:
   
    a.	Сохранить — сохраняет новое обращение при условии, что все обязательные поля заполнены и нет валидационных ошибок. 
  
    b.	Закрыть — закрывает карточку без сохранения информации.
  
    c.	Удалить — кнопка-заглушка для удаления. Функционал кнопки реализуется в другой истории (см. Задание-5). 
  
Валидации: 

•	При попытке ввести недопустимые значения в поля «Инициатор» и «Название» выводить пользователю ошибку с текстом: «Недопустимое значение поля <Название поля>». 

•	Максимальное количество символов для ввода в поле «Инициатор» - 150 символов. 

•	Максимально количество символов для ввода в поле «Описание» - 1000 символов. 

•	При попытке сохранить обращение без заполнения обязательных полей, выводить пользователю ошибку с текстом «Поле <название поля> не должно быть пустым». 

# User Story - 2: Пользователь может просматривать созданные обращения в табличном представлении.

Задачи: 
•	Реализовать страницу просмотра всех записей (табличное представление обращений). 

Требования: 

1.	Табличное представление обращений должно содержать следующий состав обязательных колонок:
   
    a.	Чек-бокс — колонка с чек-боксами, предназначенная для выбора одного или нескольких записей в табличном представлении. 

    b.	«Id» — уникальный идентификатор обращения.   

    c.	«Инициатор» — физическое лиц (пользователь), зарегистрировавшее обращение. Значение должно тянуться из поля «Инициатор» карточки обращения. 

    d.	«Название» — наименование (тема) обращения. Значение должно тянуться из поля «Название» карточки обращения. 

    e.	«Описание» — развернутое описание обращения. Значение должно подтягиваться из поля «Описание» карточки обращения. В табличном представлении описание должно быть обрезано до 300 символов. 

    f.	«Крайний срок» — Крайний срок взятия обращения в работу. Значение должно подтягиваться из поля «Крайний срок» карточки обращения и отображаться в формате дд.мм.гггг чч:мм:сс.  

    g.	«Статус» — текущий статус обращения. Значение должно подтягиваться из поля «Статус» карточки обращения. 

    h.	«Категория» — тип обращения. Значение должно подтягиваться из поля «Категория» карточки обращения. 

    i.	«Дата создания» - Дата создания обращения. Значение должно подтягиваться из поля «Дата создания» карточки обращения и отображаться в формате дд.мм.гггг чч:мм:сс.  

2.	Обращения в табличном представлении должны быть отсортированы по дате создания  от новых к старым.
   
3.	На странице с табличным представлением должны быть предусмотрены следующие кнопки управления:
   
    a.	«Создать» — кнопка предназначена для создания нового обращения. По кнопке должна открываться карточка нового Обращения. 

    b.	«Удалить» — кнопка предназначена для удаления обращения из табличного представления. 

4.	Табличное представление должно содержать вертикальный и горизонтальный скролл для прокрутки записей.

# User Story - 3: Пользователь может редактировать созданные обращения.
Задачи: 
    Обеспечить возможность открытия карточки существующего обращения для редактирования из табличного представления и по прямой ссылке. 
Сценарий: 
    Пользователь находится на странице с табличном представлении обращений или переходит на страницу обращения по прямой ссылке. 
    
Набор шагов: 

1.	Пользователь в табличном представлении нажимает левой клавишей мыши на строку записи обращения или вводит в строку Url в браузере прямую ссылку на страницу обращения.
   
2.	Система открывает карточку обращения, доступную для редактирования.

3.	Пользователь редактирует требуемые поля.

4.	При необходимости отрабатывают валидации (см. список ниже).

5.	Пользователь нажимает кнопку «Сохранить».

6.	Система сохраняет изменения.

7.	Пользователь нажимает кнопку «Закрыть».

8.	Система закрывает сохраненную карточку обращения.

    
Валидации: 

•	При попытке ввести недопустимые значения в поля «Инициатор» и «Название» выводить пользователю ошибку с текстом: «Недопустимое значение поля <Название поля>». 

•	Максимальное количество символов для ввода в поле «Инициатор»  - 150 символов. 

•	Максимально количество символов для ввода в поле «Описание» - 1000 символов. 

•	При попытке сохранить обращение без заполнения обязательных полей, выводить пользователю ошибку с текстом «Поле <название поля> не должно быть пустым». 

# User Story - 4: Пользователь может удалять созданные обращения.

**Сценарий-1**: Удаление обращения из табличного представления
Пользователь находится в табличном представлении обращений
Набор шагов: 
1.	В табличном представлении при помощи колонки «Чек-бокс» пользователь выбирает обращение, требуемое к удалению. 
2.	Система выделяет строку записи обращения. 
3.	Пользователь нажимает на кнопку «Удалить». 
4.	Система удаляет обращение из табличного представления.
   
**Сценарий-2**: Удаление обращение через карточку обращения
Пользователь находится в карточке обращения
Набор шагов: 
1.	Пользователь в карточке обращения нажимает кнопку «Удалить». 
2.	Система удаляет обращение физически из базы обращений.
3.	После удаления обращения система перенаправляет Пользователя на табличное представление обращений.  

# User Story - 5: Система может в автоматическом режиме переводить обращения в статус Просрочено. (Выполнение данной истории опционально, но желательно)
Задачи: 

•	Реализовать сервис который в автоматическом режиме переводил обращения в статус Просрочено.

Требования: 

Автоматический перевод обращения в статус Просрочено, должен происходить по истечению 24 часов, с момента создания обращения.
